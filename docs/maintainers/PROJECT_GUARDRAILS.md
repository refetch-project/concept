# Refetch Core 项目护栏与验证记录

> 维护者文档。本文不参与首页叙事，用于约束范围、记录反对意见，并在投入扩大之前判断项目是否值得继续。

## 1. 当前工作假说

Refetch Core 的最小工作假说是：

> 对相同候选集，使用用户明确选择的任务 Lens，并提供可追溯的排序理由，可以比单一的时间、热度或平台排序更快地帮助用户找到当前有价值的信息。

该假说只涉及“信息重新组织是否有用”，不预设完整的 App Semantic Contract、行业标准或跨协议生成层一定成立。

## 2. 必须保持的架构边界

### 2.1 分离五类对象

- `FeedCandidate` 保存来源事实、主体、触发事件与来源证据。
- `AnalysisRecord` 保存模型或规则产生的语义分析、置信度与版本。
- `LensProfile` 保存用户当前任务及可编辑偏好。
- `RankingDecision` 保存资格判断、得分贡献与结构化理由。
- `FeedSlate` 保存最终列表及列表层面的多样性与探索信息。

任何实现若把上述对象重新合并成一个不断增加字段的万能内容类，都需要先提交设计说明。

### 2.2 Core 保持来源无关

Core 中不应出现 `if source == github`、`if source == bilibili` 等平台分支。平台差异由 Adapter 转换，确实无法通用的字段放入带命名空间的扩展区。

### 2.3 排序理由来自真实决策轨迹

推荐理由必须引用排序中实际使用的特征、规则或证据。模型可以负责措辞，但不得在排序完成后自由生成无法追溯的因果解释。

### 2.4 Lens 表达当前任务

Lens 是用户可查看、可修改、可切换的任务视角。它不应依赖对人格、政治倾向、健康状态或其他敏感身份的隐式推断。

### 2.5 多样性属于列表

多样性、新颖性覆盖和探索比例在候选项之间计算。不要把“多样性”固化成单条内容的全局属性。

### 2.6 分析与决策分别版本化

`AnalysisRecord` 必须记录分析器、模型或规则版本。`RankingDecision` 必须记录 Lens、Policy、Core 与输入版本，以便固定样本可以重放。

## 3. 基础阶段暂不实现的内容

在第一轮验证完成前，以下工作不进入 Core：

- 视频下载、播放器、字幕抓取与完整视频转录；
- 平台登录、Cookie 管理、账号自动化与风控规避；
- 推荐模型训练、用户画像模型和长期行为预测；
- 云端用户数据仓库、跨设备同步和社交关系图；
- 插件市场与运行第三方不受信代码的沙箱；
- 同时维护多种语言的完整行为实现；
- MCP、AG-UI、A2UI、AppFunctions 或 App Intents 代码生成；
- 完整 App Semantic Contract 规范；
- 与 Feed Lab 验证无关的大规模 UI 组件库；
- 面向所有内容类型的统一“质量真值”。

这些方向可以留作后续项目或独立 RFC，但不得成为第一版 Core 的依赖。

## 4. 第一阶段范围预算

前三个月使用固定预算控制投入：

| 维度 | 上限 |
| --- | --- |
| 正式验证来源 | GitHub 与 RSS 两种 |
| 参考 Core | Dart 一种完整实现 |
| 辅助语言 | TypeScript 只负责 Schema 校验或 Feed Lab |
| 模型接入 | 一个分析器适配器，加一个无模型固定样本模式 |
| 验证宿主 | 一个 Web Feed Lab |
| Lens | 生产可用、前沿观察、贡献机会三种核心 Lens |
| 测试数据 | 50–200 个经过人工核对的候选项 |

PiliPlus/Bilibili 在 GitHub + RSS 契约稳定后进入集成验证，不与基础 Schema 并行扩张。

## 5. 可验证假设

### H1：信息发现效率

在相同候选集下，Lens Feed 找到前三个“用户认为值得进一步查看”的项目所需中位时间，应明显低于时间排序或热度排序。

首轮探索性门槛：至少 5 名目标用户、每人至少 3 次任务，时间中位数改善达到 20%。该门槛只用于决定是否继续，不作为对外产品结论。

### H2：Lens 具有真实区分度

“生产可用”“前沿观察”“贡献机会”的前十结果应存在可解释差异，用户也应能根据内容判断当前使用的是哪种 Lens。

警戒信号：不同 Lens 的 Top 10 长期重叠超过 80%，且测试用户无法描述结果目标的差异。

### H3：推荐理由可信

用户抽查推荐理由时，应能从来源证据、分析记录或确定性规则中找到对应依据。

警戒信号：超过 10% 的主要推荐理由无法回溯，或理由与实际得分贡献方向相反。

### H4：跨来源抽象成立

GitHub 与 RSS Adapter 应能输出同一组核心对象，Core 在处理两种来源时无需平台条件分支。

警戒信号：为了第二来源不断向核心对象加入只对单一平台成立的必填字段，或 Core 出现平台名称判断。

### H5：维护者愿意持续使用

Feed Lab 应首先成为维护者自己的真实工具。

警戒信号：功能可运行两周后，维护者每周主动打开少于三次，且原因是结果缺乏实际价值而不是稳定性问题。

## 6. 停止、收缩或转向条件

出现以下情况时，应暂停扩展来源和宿主，先判断是否收缩项目：

1. Lens 切换只能改变文案，无法稳定改变内容选择和排序。
2. 结构化理由比直接查看来源花费更多时间，且没有提升判断信心。
3. GitHub 与 RSS 无法共享核心对象，平台特殊字段持续侵入排序层。
4. 大部分价值实际来自摘要功能，Lens、Policy 与 Slate Selection 没有独立贡献。
5. 在固定样本上无法重放相同版本的排序结果。
6. 模型费用、延迟或数据上传成为体验成立的必要条件。
7. Feed Lab 无法在真实使用中优于保存搜索、RSS Reader 或 GitHub Notifications 的简单组合。

对应处理方式：

| 结果 | 处理 |
| --- | --- |
| 只有摘要产生价值 | 收缩为跨来源语义提取契约与本地摘要工具 |
| 只有 GitHub 场景成立 | 收缩为 GitHub Developer Intelligence 工具 |
| 只有特定宿主成立 | 保留宿主插件，停止通用 Core 叙事 |
| Contract 有价值但排序无优势 | 发布 Schema 与 fixtures，暂停推荐引擎 |
| WoT/OpenAPI 加少量扩展即可覆盖上层语义契约 | 放弃独立 App Semantic Contract，实现兼容 Profile |
| 用户不愿主动配置 Lens | 改为少量任务预设与渐进式编辑，不扩建 Persona 系统 |

## 7. 常见失败形态

### 7.1 变成“带 AI 摘要的列表”

表现：所有工程时间用于 Prompt、摘要卡片与模型切换，Lens 和决策轨迹长期停留在概念层。

纠正：固定一批人工 AnalysisRecord，优先证明不同 Lens 的确定性排序价值。

### 7.2 变成新的 GitHub 客户端

表现：大量时间进入 OAuth、通知、Issue 编辑、代码浏览与仓库管理。

纠正：Feed Lab 只读取公开或用户明确提供的数据，交互回跳来源页面。

### 7.3 过早追求通用协议

表现：在两个真实 Adapter 之前设计大量抽象实体、协议 Binding 与多语言生成器。

纠正：所有必填字段必须能由 GitHub/RSS fixtures 和具体排序需求证明。

### 7.4 分数堆叠失去含义

表现：质量、重要性、新鲜度、新颖性、多样性和 Persona 匹配被混入一个难以解释的总分。

纠正：区分来源事实、语义分析、Lens 相关性、资格规则和 Slate 约束；每个分数声明范围和计算阶段。

### 7.5 生成事后解释

表现：排序系统输出一个分数，再让 LLM 猜测“为什么推荐”。

纠正：先生成结构化特征贡献和证据引用，LLM 只负责可选的语言表达。

### 7.6 隐私承诺与实现不一致

表现：对外使用“本地优先”，实际默认把正文、浏览历史或 Lens 发送到远程模型。

纠正：每个 Analyzer 声明数据边界；远程传输采用显式配置；fixtures 与无模型模式始终可运行。

## 8. 对外叙事约束

对外优先使用以下表达：

- 用户选择的信息视角；
- 可解释、可追溯的信息组织；
- 模型无关、来源无关与本地优先；
- 同一批信息可以服务不同任务；
- 受控探索预算和列表多样性。

谨慎使用以下主张，直到有数据支持：

- “打破信息茧房”；
- “比平台推荐更好”；
- “行业标准”；
- “适用于所有应用”；
- “完全保护隐私”；
- “AI 理解了用户”。

## 9. 决策检查点

### Checkpoint A：契约进入实现前

- 至少 20 个 GitHub 与 20 个 RSS 候选样本。
- 每个核心字段能由两个真实使用案例解释。
- 三个 Lens 均有手工预期 Top 10。

### Checkpoint B：开始 Feed Lab 前

- 固定版本在相同输入下得到相同输出。
- 每个主要得分贡献具有 evidence reference。
- Core 测试中没有平台名称分支。

### Checkpoint C：开始 PiliPlus 集成前

- 至少完成 5 名目标用户的小规模任务测试。
- H1、H2、H3 没有触发停止条件。
- 维护者已连续两周将 Feed Lab 用于真实信息发现。

## 10. 维护方式

本文随关键设计决策更新。调整范围预算、验证门槛或停止条件时，应在提交中说明依据，避免在结果不理想时悄悄移动判断标准。
